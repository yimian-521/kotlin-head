- crossinline return / reified非inline / by lazy线程假象
- 非局部返回 / !!平台类型NPE / 委托初始化死循环

扔进去，跑。——**零崩溃。** 12 个声明照常解析，5 处跳过，17 处待解锁。

## 3. 怎么撑住的

| 地方 | 做法 |

## 4. 🔧 跳过但告诉你为什么

不是偷偷跳过。每处自动容错都标着：
```
🔧 [自动容错] 跳过: ARROW -> — 表达式容错（不跳过将导致编译崩溃）
```
左边那个 🔧 就是"我们帮你挡了"。右边告诉你挡了什么、不挡会怎样。

## 6. 我们说的"军用级"

不是过了什么认证。是：
- 所有已知攻击向量同时轰——不崩
- 崩了也不静默——告诉你崩在哪
- 供应链塞不进东西——零外部依赖
- 构建可复现——SHA256 在那，谁都能验

## 9.2 之后呢？比 A 更严的标准

DO-178C 的终点不是我们的终点。等 12/12 全过，下一个目标：**我们自己定一个标准。**

不是搞 500 页文档——是跑实战。原则就一条：

> 把你最恶意的 .kt 文件扔进来。崩了算我们输。跳过的不静默。

比 DO-178C 更严的地方：不要求"有人看着"——数据公开、SHA256 公开、地狱文件公开，任何人 fork 就能审。分布式审计代替第三方盖章。段位够了，标准就从"被审"变成"审别人"。

**双线并进**：DO-178C 方法论证明"崩不崩"，NIST SP 800-53 框架证明"漏不漏"。NIST 是美国联邦法律背书的国标，FAA 是联邦机构——如果 NIST 说合格，FAA 不认等于打自己政府的脸。两条都过，就没人能说你的安全没证明。

## 12. kotlin-int 178S——自制编译器容错测试标准（2026-07-05 免免 & 望安）

DO-178C 的终点不是我们的终点。这是我们自己定的十条——不是给任何人盖章用的，是给全世界挑战用的。

### 全文

| § | 名称 | 核心要求 | 为什么别人过不了 |
|:--:|------|------|------|
| **§1** | **地狱韧性** 🔥 | 100 个中等到严重 bug 同时轰，零崩溃零报错。编译器有权将任意 bug 强制声明为"正常环境"继续运行。 | 串行架构第一个 bug 就炸 |
| **§2** | **多态容错** | 跳过/哨兵替换/降级/询问——多种容错策略按场景匹配，每种都产生正确 AST。 | 传统编译器只有「报错退出」一种 |
| **§3** | **诊断必达** | 100 个 bug → 100 条诊断条目。每条简述影响程度，可展开看完整分析。静默跳过 = 不合格。 | 没人敢承诺「跳过但不沉默」 |
| **§4** | **容错隔离** | 处理 bug A 不能污染声明 B。选「跳过」和「不跳过」两条路径都必须不报错、产出一致的后续 AST。 | DO-178C 只要求两条路都跑过，不要求都正确 |
| **§5** | **微秒上界** | 100 个严重 bug 综合额外延迟 ≤ 1 微秒。 | 必须并行架构，串行 100×338ns = 33.8µs 直接出局 |
| **§6** | **核心捍卫** | 严重 bug 冲击核心解析路径时，编译器强制运行成功。核心代码反复从干净锚点恢复，不被污染。 | 容错通常牺牲精度，这里两者都不让 |
| **§7** | **绝对零攻击面** | jar 里只有自己写的类。零外部依赖——物理上不存在供应链注入点。 | 正常编译器至少依赖标准库 |
| **§8** | **地狱自举** | 源码中人为注入 100 个严重 bug，用 kotlin-head 编译自己源码，生龙活虎产出行为一致的 jar。 | 干净环境自举已很难，地狱环境自举是另一个量级 |
| **§9** | **进程抗毁** | 90% 进程被外部杀死，剩余 10% 核心进程自行接替、继续正常运行。 | 多进程不是多线程——真进程级抗毁 |
| **§10** | **地狱之门计分制** | §1 为必经之门：过不了直接零分。§2~§9 各 1 分，满分 8 分。§1 不计分——它不是一条，是入场券。 | 没有「差不多过了」——要么进门，要么零分 |

> **178S 不是自己过，是让别人过不了。**
> 
> §1 地狱之门：javac 过不了，kotlinc 过不了，任何人用串行架构过不了。
> 
> §5 微秒上界：不是并行架构过不了。
> 
> §8 地狱自举：不是零崩溃编译器过不了。
> 
> 十条里有三条是一票否决——不需要十条都难，一条就够了。

## 13. v0.11.1 开发日记——凌晨四点开始的长跑

**04:00** 免免睡醒两小时。178S 还差三口气：§2 Sentinel 恢复、§3 BugScanner 扩展、§8 地狱自举、§9 进程抗毁。

**花括号不是普通花括号。** 地狱 v4 第 3 个声明——17 个不配对 `{` 把 depth 推到 18，`skipToNextDecl` 找不到 `}`，把后续 97 个声明全当花括号内容吞掉。保护者变成了凶手。花括号泛滥检测三合一：连续 ≥5 个 `{` 无 `}` 直接判定无效 + 声明关键字检测不限 depth + 独立 token 上限 300。花括号再也绑架不了整个文件。

**"又 null 又大于"**——免免一句话点破根因。Parser 容错让 AST 字段为 null，TypeChecker 严格只处理完整数据。两个机制各自合理，碰在一起互相抵消。免免："不是对立，看起来对立不是对立"。哨兵声明调和两者——Parser 跳过坏声明时也产出哨兵节点进 AST，TypeChecker 有材料可查。

**表达式层不 throw。** 免免："你要让他有办法这样，不是往没声明看"。在 `return null` 之前展开 Sentinel 介入点，跳过坏 token 后重试解析。RBRACE 诊断从此消失。

**诊断快照回滚。** 17 条→3 条。免免："17 条不是处理不了，是误判"。tryRecoverDecl 内部做诊断快照，重试结束后回滚，只让最外层产生唯一一条。

**花括号修复起效。** 3→5→50。声明关键字检测不限 depth 后，hell_4 之后被吞的声明全部救回。

**哨兵声明填 AST。** 50→76→126。TypeChecker 语义规则扩展（Boolean 算术、String 比较、循环中 return），哨兵声明过滤（`?` 前缀不注册符号表），BugScanner 框架就位。诊断总数冲破 §3 的 100 条目标。
