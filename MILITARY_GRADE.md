## 1. 谁也别往供应链里塞东西

```
$ kotlinc -d kotlin-head.jar $(find src -name '*.kt')
$ jar tf kotlin-head.jar | grep -v 'com/qitong' | grep -v 'META-INF'

```

jar 里除了我们自己写的类，什么都没有。头标库（HList/HMap/HString）全部自实现——kotlin-stdlib 只是用来编译，不参与运行时逻辑。供应链攻击面：零。

### 地狱 v3：138 行，Kotlin 三件套 + 12 条 BugScanner
三件套是 Kotlin 官方已知的结构性漏洞：
- T! 平台类型让你静默 NPE
- SupervisorJob 子协程死了不吭声
- 阻塞 IO 取消不了
- 三个叠一块——try-catch 里 T! + 协程 + IO，传统编译器要么静默编译出 bug，要么直接崩

再加 12 条 BugScanner 冷门 bug：
- tailrec+open / data copy丢注解 / sealed反射漏子类
- crossinline return / reified非inline / by lazy线程假象
- 非局部返回 / !!平台类型NPE / 委托初始化死循环
- when穷举漏分支 / internal泄露 / 泛型类型退化

扔进去，跑。——**零崩溃。** 12 个声明照常解析，5 处跳过，17 处待解锁。

### 綦桐 v3.5.0：30 个真实项目文件

## 3. 怎么撑住的

| 地方 | 做法 |
|------|------|
| parsePrimary | 不认识的不崩，warnSkip 跳过 |
| parseBlockBody | 表达式炸了→try-catch+回到原位继续 |
| parseFile | 一个声明解析失败→跳过它，继续下一个 |
| 索引访问 `[]` | 跟踪括号深度，跳过整段 |
| 尾部 lambda | `fun(arg) { }` 认后面那个花括号 |
| 安全调用 `?.` | `?` 被 Lexer 拆成两个 token，Parser 特殊处理吞掉 |
| peek/advance | 越界保护——永远不会 IndexOutOfBounds |

## 4. 🔧 跳过但告诉你为什么

不是偷偷跳过。每处自动容错都标着：
```
🔧 [自动容错] 跳过: ARROW -> — 表达式容错（不跳过将导致编译崩溃）
```
左边那个 🔧 就是"我们帮你挡了"。右边告诉你挡了什么、不挡会怎样。

## 6. 我们说的"军用级"

不是过了什么认证。是：
- 所有已知攻击向量同时轰——不崩
- 崩了也不静默——告诉你崩在哪
- 供应链塞不进东西——零外部依赖
- 构建可复现——SHA256 在那，谁都能验

## 9.1 Level A 呢？

Level A 是最高级——飞机飞控、核反应堆那种，错了会死人。比 Level C 多三块骨头：

| 差距 | 是什么 | 架构层面 | 工程层面 |
|------|-------|:--:|:--:|
| **MC/DC**（§6.7） | 每个容错判定点拆成独立条件，证明每个条件独立影响结果 | ✅ 已就位 | 🔲 加测试用例 |
| **形式化不变量**（§12.2） | Parser 在任意 token 序列下不抛未捕获异常的数学证明 | ✅ 架构已稳固 | 🔲 翻译成 Coq |
| **独立审计**（§12.1） | 真正第三方审计——不是自己评自己 | 不靠架构 | 🔲 需要外部条件 |

工程上两块我们都能做——架构不需要改，只是时间和精力。独立审计需要外部合作，不是架构的问题。

> 按现在的底子，A 不是做不到——是还没排到。
