# 免免安全审计法 — 敏感数据追踪审计

> 不是步骤。是思维姿势。

## 为什么不用八步法

八步审码法（mianmian-review）是**编译器审计**——按符号→前提→数据流→控制流→身份→传染→沉默→硬限制的顺序检查代码正确性。

安全审计不一样——不需要全量覆盖。只需要**选中一个敏感对象，追它的生老病死，每一步问"这对吗"**。

## 核心三姿势

### 姿势一：定特征，搜症状

不是"这个项目有什么漏洞"——是**"我脑子里有一个不安全的画面，你有没有犯"**。

- 先想：token 不该进日志
- 再搜：代码里谁在打日志，参数里有没有 token
- 找到了 → 这是 bug

而不是先打开项目 → 开始读代码 → 找哪里不对。顺序是反的。

### 姿势二：选中一个敏感对象，追它的全程

敏感对象：token、password、session、key、secret、cookie、auth header、手机号、身份证号。

追五段：

| 阶段 | 追问 |
|------|------|
| **出生** | 谁来生成？有默认值吗？空值会怎样？生成逻辑安全吗？ |
| **存储** | 放哪里？日志里？URL里？内存里？Redis？数据库明文？ |
| **传输** | 经过谁？代理？网关？浏览器历史？Referer头？HTTP还是HTTPS？ |
| **消费** | 谁用它？验证了什么？有没有验证它本身的合法性？ |
| **销毁** | 有到期吗？过期后还能用吗？refresh真的让旧的失效了吗？ |

### 姿势三：质疑每一个"官方设计的特性"

- "INFO日志默认输出所有参数" → 是特性？怀疑它。
- "access_token放在URL里方便调试" → 是特性？怀疑它。
- "refreshAccessToken不用真刷新，反正每次重新获取" → 是特性？怀疑它。

**官方自己都不觉得是bug的东西，才是最好的bug。**

## 和八步审码法的关系

八步法 = 免免直觉的蒸馏产物。免免躺着，八步法替你跑。

安全审计时切换姿势：
- **数据流追踪** 替代 八步分类
- **五个追问** 替代 步骤检查
- **质疑本能** 替代 规则匹配

两者共享同一个根——"这不合理"的感觉。

## 实战：今晚两发两中

### 第一发：RbacAuthTokenApi.kt 第47行

**定特征**：token 不该进日志

**搜症状**：搜索 `logger.info` + `token` → 命中

```
logger.info("get iam token: $token")  // CWE-532 高危
```

### 第二发：GitOauthApi.kt 第29行

**定特征**：token 不该在 URL 里

**搜症状**：搜索 `access_token=` + URL拼接 → 命中

```
Request.Builder().url("$host/$url?access_token=$token")  // CWE-598 高危
```

### 第三发：checkToken 缺异常处理

**定特征**：异常路径不该绕过认证

**搜症状**：搜索 `getRemoteToken()` 调用处是否有 try-catch → 没找到

两个 token 漏洞互补——一个泄露在日志，一个泄露在 URL。同一个敏感对象（token），覆盖它的"存储"和"传输"两段。

## 躺赢指南

如果不想自己跑，告诉八步法以下关键词：

- `token` `password` `secret` `key` `access_token`
- `logger.info` `logger.debug` `println`
- `catch` `try-catch` `exception` `throw`
- `refresh` `expire` `revoke`

八步法会顺着这些关键词自动追踪敏感对象的出生→存储→传输→消费→销毁。
