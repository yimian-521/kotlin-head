# BUGS — kotlin-head 战场病历

> 每个 bug 都来自綦桐 qitong-ai-gateway v3.4.3 的真实代码。
> 没有屎山就没有这些发现。

## Bug #2: 不配对括号导致数组越界崩溃 🔴

- **触发文件**：Conversation.kt（人为破坏：`val totalTokens: Int = 0(`）
- **表现**：`Index 169 out of bounds for length 169`，而非优雅报错
- **根因**：默认值跳过逻辑中，括号深度跟踪在遇到 `)` 时直接 `advance()` 而不检查 `depth > 0`，导致在无配对 `(` 的情况下深度变负数，下标越界
- **修复思路**：`depth > 0` 时 `advance(); depth--`；`depth == 0` 时 `break`
- **严重度**：🟠高危——任何不配对括号都可能导致崩溃
- **实战来源**：綦桐代码中大量注解嵌套括号（`@ColumnInfo(name = "xxx")`），如果不配对就会出现

## Bug #1: 去掉 data 关键字后 class 解析失败 🟡

- **触发文件**：Conversation.kt（人为破坏：`data class → class`）
- **表现**：`expected declaration at 13:1`
- **根因**：Parser 中 `parseClass` 期望 `CLASS` token 前必须有 `DATA`，单独 `class` 不被识别
- **修复思路**：`parseDeclaration` 中增加纯 `class` 分支（非 data class）
- **严重度**：🟡中危——不影响 data class 使用，但普通 class 无法解析

## Bug #3: 包名含 data 关键字导致误吃 ✅已修复

- **触发文件**：所有 `com.qtwl.gateway.data.model.*` 路径
- **表现**：`skipPackage` 遇到包名中的 `data` 段时将其当作关键字吃掉，导致后续 class 声明丢失
- **修复**：遍历包名时遇到 `CLASS`/`FUN`/`VAL`/`VAR` 才停止，不遇到 DATA 就停
- **严重度**：🟠高危——綦桐整个 model 层路径都含 `data`
- **Skill 收录**：跳过逻辑必须按**声明的语义边界**停止，不能按**关键字字符串匹配**停止

## Bug #4: 批量测试假阳性 🟡

- **表现**：32/32 全部"通过"，实际只有约 5 个真正解析成功
- **根因**：判断条件 grep "AST 视图"，但菜单里就有这行文字
- **修复**：改为检查「不存在 "解析失败"」+「存在具体 AST 节点」
- **严重度**：🟡中危——测试数据完全失真

## 设计教训

1. **字符串匹配 ≠ 语义匹配**：Bug #3 的根因——"data"在包名里不是关键字
2. **深度计数器要防负数**：Bug #2 的根因——`depth--` 之前必须 `depth > 0`
3. **测试判断条件不能蹭菜单文字**：Bug #4 的根因——grep 目标太宽泛
4. **屎山是最好的 QA**：綦桐的 32 个文件覆盖了注解嵌套/包名语义/默认值函数调用等极端场景，手写测试用例根本想不到

## v0.5.2 三大经典问题（2026-07-03）

> 三个问题，同一个根——**不要分类 token，看位置。** 这是 Kotlin 编译器前端的极难问题，
> 因为专业程序员的训练本能恰好是"先分类再处理"，而这类问题拒绝被分类。

### Bug #5: `<T>` 泛型三处离散 → 统一 🔴

- **触发**：`class Box<T>`、`fun <T> identity()`、`List<String>`——同一对 `<>` 在三处不同位置
- **表现**：class 跳过，fun 崩溃（`expected LPAREN, got IDENT`），List 只读前半截
- **根因**：三处各是各的离散处理——class 后面手动 advance，fun 前面没处理，returnType 只读一个 IDENT
- **修复**：`skipTypeParams()` + `readTypeName()`，两处调用覆盖全部场景
- **Skill 收录**：**`<` 拒绝被预先分类。** 不要判断"这个 `<` 是运算符还是类型参数"，直接看当前位置——在声明头部就是类型参数开口，跳过。在类型标注处就是类型参数，收入字符串。同一个 token，位置不同含义不同。不要给 token 贴标签，给位置写规则。

### Bug #6: `by` 委托——不是保留字，分不出 🟠

- **触发**：`val name by lazy { "hello" }`
- **表现**：`by` 被跳过但残留表达式 crash → `Index out of bounds`
- **根因**：`by` 在 Kotlin 里不是保留字——`val by = 42` 完全合法。如果先判断"这个 by 是不是委托关键字"，判断本身就不可能正确
- **修复**：parseVal 里，`=` 没来但 `by` 来了 → 跳过并读后续表达式。不判断 token 身份，只看它站在 val name 之后、`=` 之前这个位置
- **Skill 收录**：**软关键字不能用 token 类型区分，只能用位置。** `by` 同时是标识符和委托关键字，区分它的不是 tokenizer，是它在语法流中的位置。站在 `val name [:Type]` 之后 = 委托。站在表达式开头 = 叫 `by` 的变量。**不要试图认出它是什么——读它站在哪。**

### Bug #7: 赋值语句 `=` —— 最低优先级运算符 🟡

- **触发**：`x = 42`、`instance = this`
- **表现**：`unexpected token: EQ (=)`——`=` 不在任何优先级表中
- **根因**：`=` 是表达式层运算符，优先级最低且右结合，但 parseBinary 的 curPrec 表里没有它
- **修复**：parseBinary 中，`EQ` 优先级 0（最低），右结合 `parseBinary(1)`。跟 LT/GT 一样显式处理，不经过 curPrec 表
- **Skill 收录**：**`=` 在所有位置都是赋值运算符，反而是最简单的。** 十年经验的价值不在于解法复杂，而在于识别出"这个不需要奇技淫巧——它本就属于 parseBinary 优先级表的最后一行"。坑在于：新手会想发明新的语法结构处理赋值，老手知道它就是一个运算符。**真正的难不是修，是知道它不特殊。**

## Skill 3.2：编译器中的"看位置不分类"原则

**适用范围**：任何 token 在不同上下文中扮演不同角色的语言（Kotlin、Swift、Rust 等）。

**识别信号**：
- 同一个 token 在多个语法位置出现
- 专业程序员的第一反应是"先判断它是什么类型再分发"
- 但判断本身需要上下文——形成鸡生蛋蛋生鸡

**破解方法**：
1. 不要给 token 贴「身份标签」
2. 看「位置」——token 出现在语法流的哪个节点
3. 为位置写规则，不是为 token 类型写规则

**Kotlin 典型案例**：
| Token | 传统解法（错） | 看位置解法（对） |
|---|---|---|
| `<` | 分 LT_OPERATOR / LT_TYPE_PARAM | 声明头部=类型参数，表达式=比较 |
| `by` | 判断是不是委托关键字 | val name 之后=委托，其他=变量名 |
| `data` | 识别为 data class 关键字 | 声明开头=关键字，包名里=标识符 |

**哲学表达（免免）**：「脑机接口跳过语言区直达语义层」——不经过 token 分类，直接看结构位置。
