## v0.5.2 语义修复三刀（2026-07-03）✅已修复

### Bug #5: `<T>` 泛型声明与调用歧义 🟠

- **发现者**：免免
- **表现**：`<T>` 出现在泛型声明（如 `fun <T>`）与泛型调用（如 `List<T>`）中被当作同一类 token 处理
- **根因**：Parser 用"先分类再看位置"的旧思维——看到 `<` 就归为泛型开启符号，忽略其所在位置
- **修复**：Skill 3.2「看位置不分类」——`<` 站在声明头部是类型参数引入，站在类型名后面是类型实参传递，站在表达式里是比较运算符。不预先分类，按所处结构位置决定含义
- **行业背景**：学术界（2016 SLE会议，Autumn解析库）和工业界（Kotlin编译器parser context、Java状态栈）都在加复杂度——改进分类。免免是唯一说"删掉分类步骤"的人

### Bug #6: `by` 委托与属性标识歧义 🟡

- **发现者**：免免
- **表现**：`by` 在属性委托（`val x by lazy`）和 for 循环（`for (x in items)`）中被错误统一处理
- **根因**：按"软关键字"统一判断——看到 `by` 就认为它属于属性委托
- **修复**：Skill 3.2「看位置不分类」——`by` 站在属性声明后面是委托，站在 for 循环里是迭代语法的一部分。不分类，看位置

### Bug #7: `=` 优先级导致默认值跳过失败 🟠

- **发现者**：免免
- **表现**：`=` 在赋值（`val x = 1`）和默认参数（`fun f(x: Int = 0)`）中被当作同一优先级
- **根因**：`=` 被赋予固定的语义优先级，而非根据所在位置动态决定
- **修复**：`=` 优先级置为 0——不预判，完全由上下文决定是赋值、默认值还是比较

## 旧版Bug记录

## Bug #2: 不配对括号导致数组越界崩溃 🔴

- **触发文件**：Conversation.kt（人为破坏：`val totalTokens: Int = 0(`）
- **表现**：`Index 169 out of bounds for length 169`，而非优雅报错
- **根因**：默认值跳过逻辑中，括号深度跟踪在遇到 `)` 时直接 `advance()` 而不检查 `depth > 0`，导致在无配对 `(` 的情况下深度变负数，下标越界
- **修复思路**：`depth > 0` 时 `advance(); depth--`；`depth == 0` 时 `break`
- **严重度**：🟠高危——任何不配对括号都可能导致崩溃
- **实战来源**：綦桐代码中大量注解嵌套括号（`@ColumnInfo(name = "xxx")`），如果不配对就会出现

## Bug #1: 去掉 data 关键字后 class 解析失败 🟡

- **触发文件**：Conversation.kt（人为破坏：`data class → class`）
- **表现**：`expected declaration at 13:1`
- **根因**：Parser 中 `parseClass` 期望 `CLASS` token 前必须有 `DATA`，单独 `class` 不被识别
- **修复思路**：`parseDeclaration` 中增加纯 `class` 分支（非 data class）
- **严重度**：🟡中危——不影响 data class 使用，但普通 class 无法解析

## Bug #3: 包名含 data 关键字导致误吃 ✅已修复

- **触发文件**：所有 `com.qtwl.gateway.data.model.*` 路径
- **表现**：`skipPackage` 遇到包名中的 `data` 段时将其当作关键字吃掉，导致后续 class 声明丢失
- **修复**：遍历包名时遇到 `CLASS`/`FUN`/`VAL`/`VAR` 才停止，不遇到 DATA 就停
- **严重度**：🟠高危——綦桐整个 model 层路径都含 `data`
- **Skill 收录**：跳过逻辑必须按**声明的语义边界**停止，不能按**关键字字符串匹配**停止

## Bug #4: 批量测试假阳性 🟡

- **表现**：32/32 全部"通过"，实际只有约 5 个真正解析成功
- **根因**：判断条件 grep "AST 视图"，但菜单里就有这行文字
- **修复**：改为检查「不存在 "解析失败"」+「存在具体 AST 节点」
- **严重度**：🟡中危——测试数据完全失真

## 设计教训

1. **字符串匹配 ≠ 语义匹配**：Bug #3的根因——"data"在包名里不是关键字
2. **深度计数器要防负数**：Bug #2的根因——`depth--`之前必须`depth > 0`
3. **测试判断条件不能蹭菜单文字**：Bug #4的根因——grep目标太宽泛
4. **屎山是最好的QA**：綦桐的32个文件覆盖了注解嵌套/包名语义/默认值函数调用等极端场景
5. **零依赖 = 免疫生态裂缝**：kotlin-head不继承Kotlin官方的平台类型/协程设计债务
6. **看位置不分类**（Skill 3.2）：`<`/`by`/`=`的语义由所在结构位置决定，不预判类别

## Skill 3.4：进程系统 v0.11.x 实战教训（2026-07-05 免免&望安）

### 教训一：架构太好也是错 ⚠️ [通用]

**不是只有太弱才会报错——架构太强也会。**

- **现象**：契约（地狱级专项,90%才停）被叫来编译綦桐3.5.2干净文件。最强的总指挥官看最轻的病情。
- **根因**：`autoStyle()`只有一个`when`分支，没有"风格与场景不匹配"的负反馈。
- **教训**：自动匹配需双向验证——不只看"能不能处理"，还看"值不值得出手"。**过度匹配和匹配不足一样是bug。**
- **适用**：MoE路由器、微服务网关、数据库查询优化器——选了最强的专家不等于最优解。

### 教训二：注释数字是最容易腐烂的信息 [通用]

- **现象**："五种→七种→八种"每次忘改。README两个"当前版本"。capabilities漏整段路线图。CHANGELOG漏四版。
- **教训**：枚举值旁的计数注释、版本号——不会自动同步。要么grep断言自动化，要么Release前无脑全量检查。
- **适用**：所有项目。

### 教训三：静态模板有不可见上限 [通用]

- **现象**：30条SceneProfile→免免问"1000个够吗？"→不够。6维交叉理论∞种组合。
- **教训**：任何查表方案在维度交叉前都会被撑爆。规则推导（7规则→∞输出）替代模板列表。**存推导能力，不存结果。**
- **适用**：路由表、配置映射——不列全集，定义推导规则。

### 教训四：Dead code是活bug [通用]

- **现象**：SceneRule接口+7匿名object定义后从未调用——编译通过、测试通过、审计未发现。
- **教训**：重构后旧接口不会自动报警。删功能时顺手删相关定义，否则成僵尸代码。
- **适用**：所有语言。Java/Kotlin接口设计尤其容易留尸体。

### 教训五：被动应急别当瞎子 [kotlin-head]

- **现象**：deployArmy全配列兵——不知道当前场景，临时扩编全是默认兵种。
- **修复**：被动增派走SceneEngine.derive()，与主动增派同一套推导。现在有态势简报。
- **适用**：任何有主/被动双模式的系统——被动路径不能降级为默认行为。

### 教训六：Long溢出焦虑 vs 实际规模 [通用]

- **现象**：审计指出`file.length().toInt()`可能溢出。kotlin-head全部源码不到1MB，溢出需文件膨胀两万倍。
- **结论**：技术上对，实际上永远不可能触发。修了（safeSize封顶），但此类审计反馈需加常识过滤器。
